Project Update: Unsupervised Anomaly Detection Pipeline

System: JLab Compute Cluster Digital Twin | Date: May 19 - May 23, 2023

Core Objective: Detect an unlabelled, "unknown" infrastructure anomaly that occurred on May 23 using purely unsupervised machine learning models trained on normally-behaving cluster data.

1. Critical Fix: Leakage-Free Data Splitting

A standard 70/15/15 random split was initially used, which fundamentally corrupted the pipeline. We have entirely reworked the data formulation to respect the temporal, unsupervised nature of the problem.

Design Aspect Previous Approach (Flawed) Current Approach (Corrected)
Data Split Strategy 70% Train / 15% Val / 15% Test (Random) Strict Chronological Split (No Val set)
Training Set Contained May 23 anomalous data May 19 - May 22 Only (Strictly normal data)
Testing Set Mixed normal and anomalous data May 23 Only (Contains the anomaly)
Normalization Global (Leaked future stats to train) Scalars computed exclusively on Training Set

2. Data Quality Discovery: The "CPU Data" Anomaly

During the chronological splitting rework, the pipeline uncovered a critical data collection failure in the raw Prometheus dataset.

Finding: There is an 84% scraping gap for cpu_data and disk_data leading up to the May 23 incident. The resulting Test Splits (May 23) for these modalities contain 0 rows.

Decision: We have excluded CPU and Disk metrics from model evaluation, as they cannot be used to detect the target event. We are successfully proceeding with memory_data (1.8M train rows) and slurm_data (3.7M train rows).

4. Feature Engineering Formulations

To give the models contextual awareness of time and state transitions, new features were synthesized before training.

  • Memory Aggregations & Lags (26 features): We generated rolling windows (5min, 15min, 60min) computing mean, std, max, and min. We also introduced lag1_diff to track abrupt memory spikes.
  • Slurm State Matrix (14 features): Rather than basic CPU counts, we transformed Slurm string logs into boolean state matrices (status_draining, status_down, etc.) and computed a status_transition_rate_15min to track node instability over time.

5. Baseline Modeling & "Ground Truth" Validation

Because the anomaly is unlabelled, we used Slurm Distress States (nodes reporting as DOWN, FAILING, or DRAINED) as a pseudo-ground-truth to validate if our unsupervised models were identifying real issues.

Autoencoder Breakthrough: The Autoencoder trained on Slurm transition features yielded exceptional precision. It flagged only 0.8% of the May 23 test data as anomalous, but 45.09% of those flagged points perfectly overlapped with nodes falling into distress states.

Hardware Heatmap findings:

Based on the Autoencoder reconstructions, the anomaly heavily impacted specific localized server racks. farm18 and farm14 hardware configurations accounted for over 75% of all flagged anomalous behaviors during the May 23 event.

6. Next Steps for Implementation

  1. Multimodal Ensembling: Join the Memory latent-space reconstruction errors with the Slurm latent-space errors on a unified 1-minute temporal axis to calculate a single cluster health score.
  2. Threshold Tuning: Adjust Autoencoder Sigma standard deviation thresholds dynamically based on time-of-day traffic.
